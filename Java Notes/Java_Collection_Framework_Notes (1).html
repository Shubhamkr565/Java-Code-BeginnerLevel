    String city = itr.next();

вқ— Fail-Fast Behavior
If you modify a collection directly while iterating with enhanced for-loop вҶ’ ConcurrentModificationException!

Safe way: always use itr.remove() (never list.remove() inside a for-each)

Removing while iterating with normal for-loop вҖ“ Gotcha!

// вқҢ PROBLEM: after removing "b", "c" shifts left вҶ’ skipped!

    if(al.get(i).equals("b")) al.remove(i);
}

// вң… Fix 1: Decrement i after remove

// вң… Fix 2: Traverse backward
for(int i=al.size()-1; i>=0; i--) { ... }

// вң… Fix 3: Use Iterator (best practice)

Limitations of Iterator

  • вқҢ One-direction only (forward)
  • вқҢ Only read + remove (no add, no set/replace)

17. рҹ§¬ Generics вҖ“ Deep Dive

вӯҗ What are Generics? (Java 1.5)
Write ONE class/method that works with any data type.
Think of <T> as a placeholder for a type вҖ“ decided at use-time.

Problem вҶ’ Solution

вқҢ Without Generics
class MyGen {
  Object obj;
  Object get() { return obj; }
}
// Need downcasting:
String s = (String) gen.get(); 
// Risk: ClassCastException!
вң… With Generics
class MyGen<T> {
  T obj;
  T get() { return obj; }
}
// No casting needed:
MyGen<String> g = new MyGen<>();
String s = g.get();  // вң… Clean!

Advantages of Generics

  • вң… Type safety at compile time
  • вң… No downcasting / ClassCastException
  • вң… Code reusability вҖ“ write once, use for any type
  • вң… Cleaner code
  • вҡ пёҸ Works only with reference types (objects, NOT primitives)
вқ— List<int> вҶ’ вқҢ INVALID
List<Integer> вҶ’ вң… VALID

18. рҹ”’ Bounded Types (Generics Constraints)

вӯҗ Use "extends" keyword to restrict type parameter
UNBOUNDED: class Test<T> вҶ’ Any type allowed UPPER BOUND: class Test<T extends X> вҶ’ Only X or subclasses of X For Interface: class Test<T extends Runnable> вҶ’ implementations of Runnable

Examples

class Bill<T extends Payment> { }
// Valid:   Bill<Payment>, Bill<CashPayment>, Bill<CardPayment>
// Invalid: Bill<Integer>    вҶҗ not related to Payment

class Test<T extends Number> { }
// Valid:   Test<Number>, Test<Integer>, Test<Float>
// Invalid: Test<String>    вҶҗ String is not a Number

Multiple Bounds (using &)

class Test<T extends Number & Serializable> { }
// T must extend Number AND implement Serializable
вқ— Important Rules for Bounded Types
  • вқҢ implements NOT allowed вҶ’ use extends for both class and interface
  • вқҢ super NOT allowed at class level
  • вқҢ Two class types together: <T extends Number & Thread> вқҢ
  • вң… One class + multiple interfaces: <T extends Number & Runnable> вң…
  • вң… Class MUST come BEFORE interface in multiple bounds

21. рҹғҸ Wildcards вҖ“ ?, extends, super

вӯҗ PECS Principle (Producer Extends, Consumer Super)
Producer (you read/get values) вҶ’ use ? extends T
Consumer (you write/add values) вҶ’ use ? super T

1. Upper Bound Wildcard вҖ“ ? extends T

Read-only (Covariance) вҖ“ "T or any subtype of T"
public static void print(List<? extends Employee> list) {
    for(Employee e : list) { ... }  // вң… reading OK
    // list.add(new Employee());   // вқҢ Cannot add
    list.add(null);                // вң… Only null is ok
}
Can call with: List<Employee>, List<Developer>, List<Manager>
Cannot add because compiler doesn't know the exact subtype!

2. Lower Bound Wildcard вҖ“ ? super T

Write-friendly (Contravariance) вҖ“ "T or any supertype of T"
public static void addNumbers(List<? super Integer> list) {
    for(int i=1; i<=10; i++) list.add(i);  // вң… write OK
    Object obj = list.get(0);               // вң… only Object for read
    // Integer x = list.get(0);            // вқҢ restricted
}
Can call with: List<Integer>, List<Number>, List<Object>
Cannot read as specific type вҖ“ only as Object

3. Unbounded Wildcard вҖ“ ?

"Any type" вҖ“ Use when type doesn't matter
public static void fun(List<?> list) {
    System.out.println(list.size()); // вң… size() doesn't need type
    list.clear();                    // вң… clear() doesn't need type
    for(Object obj : list) { ... }  // вң… read as Object
    // list.add("Hello");           // вқҢ cannot add
    list.add(null);                 // вң… only null
}

Summary: Wildcard Cheat Sheet

Wildcard | Reading | Writing | Use Case в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ List<T> | T type | T type | When you know type exactly List<?> | Object only | null only | Type doesn't matter List<? extends T> | T type | null only | Reading (Producer) List<? super T> | Object only | T & subtypes | Writing (Consumer)
вқ— Cannot combine both bounds!

22. рҹ§№ Type Erasure

вӯҗ What is Type Erasure?
Compiler removes generic type information at compile time.
JVM never sees generics вҖ“ it only sees raw types!

Why? Java added generics in 1.5 without changing the JVM вҶ’ backward compatibility!

How Type Erasure Works

Source Code (with Generics)After Erasure (what JVM sees)
class Box<T> { T value; }class Box { Object value; }
class Box<T extends Number> { T value; }class Box { Number value; }
List<String> list = new ArrayList<>();List list = new ArrayList();

Key Effects of Type Erasure

вқҢ Effect 1: Cannot Overload with Different Generic Types
void fun(List<Integer> list) {}
void fun(List<String> list)  {} // вқҢ Compile error!
// After erasure, both become: void fun(List list) вҶ’ CLASH
вӯҗ Effect 2: Compiler Auto-Inserts Casts
// Your code:
String s = slist.get(0);

// Compiler adds cast internally:
String s = (String) slist.get(0);  // auto-inserted!
рҹҺҒ Gift Box Analogy:
At compile time вҶ’ box has label (String, Integer)
At runtime вҶ’ label removed, only box remains (no type info!)
вҡЎ Key Takeaway
Generics provide compile-time type safety, NOT runtime safety.
At runtime, generics are gone вҖ“ JVM works with raw types only.
