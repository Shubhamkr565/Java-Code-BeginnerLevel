1. 🔗 LinkedList

⭐ LinkedList = Doubly Linked List implementing List + Deque
  • Implements both List and Deque interfaces
  • Best for frequent insertions/deletions at beginning or middle
  • Duplicates ✅ | Null ✅ | Insertion order preserved ✅
  • No index-based O(1) access (must traverse)

Internal Structure

Doubly Linked List – each node has 3 parts: null ← [ prev | data | next ] ↔ [ prev | data | next ] ↔ [ prev | data | next ] → null Node 1 Node 2 Node 3 Head ↑ Tail ↑ head points to first node, tail points to last node.

ArrayList vs LinkedList – When to use what?

OperationArrayListLinkedListWinner
get(index)O(1) ⚡O(n)ArrayList ✅
add at endO(1) amortizedO(1)Tie ~
add at start / middleO(n) (shift)O(1) (relink)LinkedList ✅
remove at start / middleO(n) (shift)O(1) (relink)LinkedList ✅
MemoryLess (contiguous)More (node + 2 pointers)ArrayList ✅
IterationFaster (cache-friendly)SlowerArrayList ✅
⭐ Rule of Thumb
  • Lots of get(index) → use ArrayList
  • Lots of insert/delete at start/middle → use LinkedList
  • Need it as a Queue or Stack → use LinkedList / ArrayDeque

LinkedList as a Queue / Stack

LinkedList<String> ll = new LinkedList<>();

// Queue operations (FIFO)
ll.offer("A");     // add to tail
ll.offer("B");
String head = ll.poll();   // remove from head → "A"
String peek = ll.peek();   // look at head without removing → "B"

// Stack operations (LIFO)
ll.push("X");      // add to head
ll.push("Y");
String top = ll.pop();     // remove from head → "Y"

❗ getFirst() throws NoSuchElementException if empty
peekFirst() returns null if empty → safer choice!

2. 🧱 Vector & Stack (Legacy Classes)

⭐ Vector – Thread-safe ArrayList (Legacy)
  • Introduced in Java 1.0 (before Collection Framework!)
  • Implements List interface
  • Every method is synchronized → thread-safe ✅
  • Default capacity = 10
  • Growth: doubles capacity (× 2) unlike ArrayList (× 1.5)
  • Slower than ArrayList (synchronization overhead)

Stack – LIFO Data Structure (Legacy)

⭐ Stack extends Vector
LIFO = Last In, First Out
Introduced in Java 1.0
LIFO – Stack Visualization: push("A") → push("B") → push("C") ┌───┐ ← TOP (last pushed, first popped) │ C │ ├───┤ │ B │ ├───┤ │ A │ └───┘ ← BOTTOM pop() → C (removes C) peek() → B (just looks, doesn't remove)

Stack Methods

MethodDescriptionThrows if empty?
push(e)Add to top–
pop()Remove & return top✅ EmptyStackException
peek()View top without removing✅ EmptyStackException
isEmpty()Is stack empty?–
search(o)1-based position from top–1 if not found
Stack<Integer> stack = new Stack<>();
stack.push(10);
stack.push(20);
stack.push(30);

System.out.println(stack.peek()); // 30 (top, not removed)
System.out.println(stack.pop());  // 30 (removed)
System.out.println(stack.pop());  // 20
System.out.println(stack);        // [10]
❗ Stack is a legacy class. Modern Java prefers:
Deque<T> stack = new ArrayDeque<>();
(ArrayDeque is faster and not synchronized)

4. 🟡 HashSet

⭐ HashSet – Fastest Set Implementation
  • Backed internally by a HashMap
  • No duplicates ✅ | Null allowed ✅ (only one null)
  • No guaranteed order – elements can appear in any order
  • All operations O(1) average: add, remove, contains
  • Default capacity = 16, Load factor = 0.75

How HashSet ensures uniqueness?

When you add an object to HashSet: 1. Calculate hashCode() of the object 2. Find the bucket (index) in internal array 3. Check if any object in that bucket is equal via equals() 4. If equal → DUPLICATE → don't add (return false) 5. If not equal → add to bucket hashCode() → finds bucket (fast!) equals() → confirms uniqueness
❗ Override hashCode() AND equals() for custom objects!
If you want HashSet to detect duplicates in your own classes (like Student), you MUST override both hashCode() and equals() methods.

Rule: If a.equals(b) is true → a.hashCode() == b.hashCode() must also be true. 
HashSet<String> set = new HashSet<>();
set.add("Apple");
set.add("Banana");
set.add("Apple");   // ← duplicate! silently ignored
set.add(null);

System.out.println(set);      // [null, Banana, Apple] (any order)
System.out.println(set.size()); // 3 (not 4!)

Iterating a HashSet

// Using enhanced for loop
for(String s : set) {
    System.out.println(s);
}

// Using Iterator
Iterator<String> itr = set.iterator();
while(itr.hasNext()) {
    System.out.println(itr.next());
}

// Using forEach (Java 8+)
set.forEach(s -> System.out.println(s));
⭐ Load Factor 0.75 means: when 75% of capacity is filled → HashSet resizes (doubles bucket count)

5. 🟢 LinkedHashSet

⭐ LinkedHashSet = HashSet + Insertion Order Preserved
  • Child class of HashSet
  • Backed by LinkedHashMap internally
  • No duplicates ✅ | Null ✅ | Maintains insertion order ✅
  • Slightly slower than HashSet (extra linked list overhead)
  • Use when you need: uniqueness + insertion order
LinkedHashSet<String> lhs = new LinkedHashSet<>();
lhs.add("Banana");
lhs.add("Apple");
lhs.add("Mango");
lhs.add("Apple");    // duplicate – ignored

System.out.println(lhs); // [Banana, Apple, Mango] ← insertion order!
// HashSet would print in any order

6. 🌳 TreeSet

⭐ TreeSet = Sorted Set (Natural or Custom order)
  • Implements NavigableSet → SortedSet → Set
  • Backed by a Red-Black Tree (self-balancing BST)
  • Elements stored in sorted ascending order by default
  • No duplicates ✅ | Null ❌ (NullPointerException!)
  • All operations: O(log n)
  • Elements must be Comparable (or provide Comparator)
TreeSet<Integer> ts = new TreeSet<>();
ts.add(50);
ts.add(10);
ts.add(30);
ts.add(20);
ts.add(10);   // duplicate – ignored

System.out.println(ts); // [10, 20, 30, 50] ← always sorted!

Extra Methods in TreeSet (NavigableSet)

MethodDescription

last()Largest element
headSet(e)Elements strictly less than e
tailSet(e)Elements greater than or equal to e
subSet(from, to)Elements from (inclusive) to (exclusive)
floor(e)Largest element ≤ e
ceiling(e)Smallest element ≥ e
higher(e)Strictly greater than e
lower(e)Strictly less than e
pollFirst()Remove and return smallest
pollLast()Remove and return largest
TreeSet<Integer> ts = new TreeSet<>(List.of(10, 20, 30, 40, 50));

System.out.println(ts.first());          // 10
System.out.println(ts.last());           // 50
System.out.println(ts.headSet(30));      // [10, 20]
System.out.println(ts.tailSet(30));      // [30, 40, 50]
System.out.println(ts.subSet(20, 40));   // [20, 30]
System.out.println(ts.floor(35));        // 30
System.out.println(ts.ceiling(35));      // 40

7. 🚶 Queue Interface

⭐ Queue = FIFO (First In, First Out)
Child of Collection. Process elements in order they were added.
Real life: Bank queue, Printer queue, CPU scheduling
FIFO – Queue Visualization: Enqueue (add) → [C][B][A] → Dequeue (remove) new elements oldest element join at rear leaves at front add("A") → add("B") → add("C") Front: A → remove() → Front: B → remove() → Front: C

Queue Methods – Two Styles!

OperationThrows ExceptionReturns null/false (safe)
Add elementadd(e)offer(e) ← preferred
Remove headremove()poll() ← preferred
View headelement()peek() ← preferred
⭐ Always prefer offer(), poll(), peek() over add(), remove(), element() – safer (no exceptions on empty queue)!

8. 🏆 PriorityQueue

⭐ PriorityQueue = Elements served by PRIORITY, not FIFO
  • Implements Queue interface
  • By default: Min-Heap (smallest element has highest priority)
  • Elements ordered by natural ordering (Comparable) or custom Comparator
  • Null ❌ not allowed
  • Not sorted! Only the head (peek) is always the minimum
  • add/offer/remove: O(log n) | peek: O(1)
PriorityQueue<Integer> pq = new PriorityQueue<>(); // Min-Heap
pq.offer(50);
pq.offer(10);
pq.offer(30);
pq.offer(20);

System.out.println(pq.peek());  // 10 (minimum always at head)
System.out.println(pq.poll());  // 10 (removed)
System.out.println(pq.poll());  // 20
System.out.println(pq.poll());  // 30  ← always gives sorted order!

Max-Heap (reverse order)

PriorityQueue<Integer> maxPQ = new PriorityQueue<>(Comparator.reverseOrder());
maxPQ.offer(10); maxPQ.offer(50); maxPQ.offer(30);
System.out.println(maxPQ.poll()); // 50 (maximum first)

9. ↔️ Deque & ArrayDeque

⭐ Deque = Double-Ended Queue (insert/remove from BOTH ends)
  • Extends Queue interface
  • Can be used as Queue (FIFO) or Stack (LIFO)
  • ArrayDeque = fastest implementation (no null allowed)
  • Preferred over both Stack class and LinkedList for stack/queue use
Deque (Double-Ended Queue): addFirst ← [D][C][B][A] → addLast pollFirst → ← pollLast You can add/remove from BOTH front and back!
OperationFront (First)Back (Last)
AddaddFirst(e) / offerFirst(e)addLast(e) / offerLast(e)
RemoveremoveFirst() / pollFirst()removeLast() / pollLast()
PeekpeekFirst() / getFirst()peekLast() / getLast()
Stack: push/poppush(e) = addFirstpop() = removeFirst
Deque<String> dq = new ArrayDeque<>();

// As Stack (LIFO):
dq.push("A"); dq.push("B"); dq.push("C");
System.out.println(dq.pop()); // C (last in, first out)

// As Queue (FIFO):
Deque<String> queue = new ArrayDeque<>();
queue.offer("A"); queue.offer("B"); queue.offer("C");
System.out.println(queue.poll()); // A (first in, first out)

11. 🟠 HashMap

⭐ HashMap – Fastest Map, No guaranteed order
  • Key uniqueness ✅ | Null key ✅ (ONE null key) | Null values ✅ (multiple)
  • No order guarantee – elements appear in any order
  • Operations: O(1) average for put, get, remove
  • Default capacity = 16, Load factor = 0.75
  • Internally uses array of buckets (LinkedList / Red-Black Tree per bucket)
  • Not thread-safe

How HashMap works internally

HashMap Internal Structure: put("name", "Alice"): 1. hashCode("name") → hash value 2. index = hash % capacity → finds bucket 3. If bucket empty → store entry directly 4. If bucket has entries → check equals() for key - Key exists → UPDATE value - Key doesn't exist → ADD to bucket (LinkedList/TreeNode) ┌────┬─────────────────────┐ │ 0 │ null │ │ 1 │ [age=25] │ │ 2 │ [name=Alice] │ │ 3 │ null │ │ 4 │ [city=Delhi]-[country=India] ← collision (linked) └────┴─────────────────────┘ (Java 8+: if bucket has 8+ entries → converts to Red-Black Tree)
HashMap<String, Integer> map = new HashMap<>();
map.put("Alice", 90);
map.put("Bob", 85);
map.put("Charlie", 92);
map.put("Alice", 95);   // key "Alice" already exists → UPDATES value

System.out.println(map);              // {Bob=85, Charlie=92, Alice=95} (any order)
System.out.println(map.get("Bob"));   // 85
System.out.println(map.containsKey("Dave")); // false
System.out.println(map.getOrDefault("Dave", 0)); // 0

Iterating HashMap

// Method 1: keySet() – get all keys, then get values
for(String key : map.keySet()) {
    System.out.println(key + " → " + map.get(key));
}

// Method 2: entrySet() – most efficient ⭐
for(Map.Entry<String, Integer> entry : map.entrySet()) {
    System.out.println(entry.getKey() + " → " + entry.getValue());
}

// Method 3: forEach (Java 8)
map.forEach((k, v) -> System.out.println(k + " → " + v));

// Method 4: values() – just values
for(Integer val : map.values()) {
    System.out.println(val);
}

15. ⚖️ Comparable Interface (Natural Sorting)

⭐ Comparable = "This class knows how to compare itself"
  • Package: java.lang
  • Contains ONE method: int compareTo(T o)
  • Used for natural ordering (default sort behavior)
  • Already implemented by: String, Integer, Double, Date, etc.
  • You implement it in YOUR custom class

compareTo() Return Values

compareTo(o) returns: Negative (< 0) → this comes BEFORE o (this is "smaller") Zero (= 0) → this EQUALS o Positive (> 0) → this comes AFTER o (this is "bigger")
class Student implements Comparable<Student> {
    int roll;
    String name;
    double marks;

    // Sort students by marks (ascending)
    @Override
    public int compareTo(Student other) {
        // Positive → this after other | Negative → this before other
        return Double.compare(this.marks, other.marks);
        // Or: return (int)(this.marks - other.marks);
    }
}

List<Student> list = new ArrayList<>();
list.add(new Student(1, "Alice", 90));
list.add(new Student(2, "Bob", 75));
list.add(new Student(3, "Charlie", 85));

Collections.sort(list);  // uses compareTo() ← natural order
// Result: sorted by marks: Bob(75), Charlie(85), Alice(90)

16. 🔧 Comparator Interface (Custom Sorting)

⭐ Comparator = "External sorting logic – flexible and multiple!"
  • Package: java.util
  • Contains ONE method: int compare(T o1, T o2)
  • Used for custom/multiple sort strategies
  • Does NOT modify the class itself → can define many comparators
  • Pass to Collections.sort(list, comparator) or list.sort(comparator)

compare() Return Values

compare(o1, o2) returns: Negative → o1 comes BEFORE o2 Zero → o1 EQUALS o2 (for ordering) Positive → o1 comes AFTER o2
// Comparator 1: Sort by name (alphabetical)
Comparator<Student> byName = new Comparator<Student>() {
    @Override
    public int compare(Student s1, Student s2) {
        return s1.name.compareTo(s2.name);
    }
};

// Comparator 2: Sort by marks descending
Comparator<Student> byMarksDesc = (s1, s2) ->
    Double.compare(s2.marks, s1.marks);  // note: s2 first for descending!

// Using comparators:
Collections.sort(list, byName);       // sort by name
Collections.sort(list, byMarksDesc);  // sort by marks desc

// Java 8 style (Comparator.comparing):
list.sort(Comparator.comparing(s -> s.name));
list.sort(Comparator.comparingDouble(s -> s.marks));
list.sort(Comparator.comparingDouble((Student s) -> s.marks).reversed());

Chaining Comparators (thenComparing)

// Sort by marks DESC, then by name ASC (if marks equal)
Comparator<Student> combined =
    Comparator.comparingDouble((Student s) -> s.marks)
              .reversed()
              .thenComparing(s -> s.name);

list.sort(combined);

18. 🧭 How to Choose the Right Collection?

START: What do you need? │ ├── KEY-VALUE pairs? │ ├── Need sorted keys? → TreeMap │ ├── Need insertion order? → LinkedHashMap │ └── Just fast lookup? → HashMap ⭐ (default) │ ├── SINGLE objects (no key-value)? │ │ │ ├── ALLOW DUPLICATES? │ │ ├── Need index access / frequent get? → ArrayList ⭐ │ │ ├── Frequent insert/delete at front? → LinkedList │ │ ├── Need FIFO? → ArrayDeque │ │ ├── Need LIFO (Stack)? → ArrayDeque │ │ └── Need priority order? → PriorityQueue │ │ │ └── NO DUPLICATES (unique only)? │ ├── Need sorted order? → TreeSet │ ├── Need insertion order? → LinkedHashSet │ └── Just fast unique? → HashSet ⭐ (default) │ └── THREAD-SAFE needed? ├── List → Collections.synchronizedList() or CopyOnWriteArrayList ├── Map → ConcurrentHashMap (best!) └── Set → Collections.synchronizedSet()

19. ⚡ Final Rapid Revision – Both Parts Combined

🚀 One-Line Mnemonics
  • 🟢 ArrayList = Dynamic array. Best for GET. Worst for INSERT middle.
  • 🔗 LinkedList = Chain of nodes. Best for INSERT/DELETE. Worst for GET.
  • 🧱 Vector = Synchronized ArrayList. Legacy. Use CopyOnWriteArrayList instead.
  • 📦 Stack = Legacy LIFO. Use ArrayDeque instead.
  • 🟡 HashSet = Unique, No order. Fastest. Uses hashCode + equals.
  • 🟢 LinkedHashSet = Unique + Insertion order. = HashSet + LinkedList.
  • 🌳 TreeSet = Unique + Sorted. No null. Uses Red-Black Tree.
  • 🚶 PriorityQueue = Min-Heap. Not sorted, but poll() always gives minimum.
  • ↔️ ArrayDeque = Double-ended. Best Stack AND Queue implementation.
  • 🟠 HashMap = Key-Value, No order. 1 null key. Fastest map.
  • 🟢 LinkedHashMap = HashMap + Insertion order. LRU Cache use case.
  • 🌳 TreeMap = Sorted keys. No null key. O(log n).
  • ⚖️ Comparable = Natural order. Inside class. compareTo()
  • 🔧 Comparator = Custom order. Outside class. compare(). Multiple strategies!
  • 🧬 Generics = Type safety at compile time. Type erasure at runtime.
  • 🃏 Wildcards: extends=read | super=write | ?=any | PECS!
❗ Common Mistakes to Avoid
  • ❌ Using == to compare Strings in collections – always use equals()
  • ❌ Modifying collection inside for-each loop → ConcurrentModificationException
  • ❌ Adding null to TreeSet/TreeMap key → NullPointerException
  • ❌ Using primitives directly in generics → use wrapper classes
  • ❌ Forgetting to override both hashCode() AND equals() for HashSet custom objects
  • ❌ Using Stack class in new code → use ArrayDeque
  • ❌ Using Hashtable in new code → use HashMap or ConcurrentHashMap
  • ❌ Assuming HashMap iteration order is consistent → it is not!
