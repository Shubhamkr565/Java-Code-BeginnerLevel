☕ Covariance & Wildcards in Java
Covariance — The Basics
  • A superclass reference can hold a subclass object → is-a relationship.
  • Arrays are covariant in Java — you can assign a subtype array to a supertype array reference.
String[] fruits = {"Apple", "Mango"}; Object[] objArr = fruits; // ✔ Allowed at compile time objArr[0] = new A(); // ✘ ArrayStoreException at RUNTIME!
⚠ Heap Pollution: Arrays allow this at compile time but throw ArrayStoreException at runtime. Compiler cannot catch it!
Generics are Invariant by Default
  • Generics do NOT follow parent-child relationship automatically.
List<String> sList = new ArrayList<>(); List<Object> oList = sList; // ✘ Compilation Error! List<Employee> emp = ...; printArray(emp); // ✔ works List<Developer> dev = ...; printArray(dev); // ✘ Error — even though Developer extends Employee
Arrays (Covariant) Generics (Invariant by default) ──────────────────── ────────────────────────────────── Object[] ← String[] ✔ List<Object> ← List<String> ✘ runtime check needed compile-time error — SAFE!
Wildcards — Making Generics Flexible
  • Use wildcards (?) to allow collections to behave like covariant/contravariant types.
? extends T
Producer — reads/gets values
Upper Bound Wildcard
→ Covariance
? super T
Consumer — writes/puts values
Lower Bound Wildcard
→ Contravariance
✦ PECS Principle:
Producer Extends — if you only READ from a source, use ? extends T
Consumer Super — if you only WRITE to a destination, use ? super T

☕ Upper Bound & Lower Bound Wildcards
1. Upper Bound — ? extends T
  • Accepts T or any subtype of T.
  • Use when you want to read from the collection.
  • Writing is NOT allowed (except null).
public static void printArray(List<? extends Employee> list) { for(Employee e : list) System.out.println(e); // ✔ Reading OK // list.add(new Employee()); // ✘ Cannot add! list.add(null); // ✔ Only null allowed } // Valid calls: printArray(new ArrayList<Employee>()); // ✔ printArray(new ArrayList<Developer>()); // ✔ printArray(new ArrayList<Manager>()); // ✔
Why can't we add? ? could be List<Developer> or List<Manager> — we can't safely add a Developer to a List<Manager>!
? extends Employee: Object | Employee ← upper bound (max) / \ Developer Manager ↑ ↑ can read can read both into Employee reference
2. Lower Bound — ? super T
  • Accepts T or any supertype of T.
  • Use when you want to write to the collection.
  • Reading is restricted — only Object type allowed on reads.
public static void addNumbers(List<? super Integer> list) { for(int i = 1; i <= 10; i++) list.add(i); // ✔ Writing OK Object obj = list.get(0); // ✔ Only Object allowed on read // Integer i = list.get(0); // ✘ Not allowed } // Valid calls: addNumbers(new ArrayList<Integer>()); // ✔ addNumbers(new ArrayList<Number>()); // ✔ addNumbers(new ArrayList<Object>()); // ✔ // addNumbers(new ArrayList<String>()); // ✘ Invalid
? super Integer: Object ← can store Integer | Number ← can store Integer | Integer ← lower bound (T itself) | (no subtypes matter here) Write Integer → safe into all above ✔ Read → only Object guaranteed ✔
WildcardReadWriteUse When
? extends T✔ as T✘ (null only)Reading / producing values
? super T✔ as Object only✔ T & subtypesWriting / consuming values
? (unbounded)✔ as Object only✘ (null only)Object methods / size/clear
⚠ Rule: You can specify EITHER upper bound OR lower bound — NOT both at the same time.
<? extends T super X> → Invalid!

☕ Unbounded Wildcard & List<?> vs List<T>
Unbounded Wildcard — List<?>
  • ? means any type — you don't care what's inside.
  • Use when method only needs Object class methods (toString(), equals()).
  • Or when using type-independent operations: size(), clear(), isEmpty().
public static void fun1(List<?> list) { for(Object obj : list) System.out.println(obj.toString()); // ✔ Object methods ok // list.add(10); // ✘ Cannot add list.add(null); // ✔ Only null } public static void fun2(List<?> list) { System.out.println(list.size()); // ✔ No type needed list.clear(); // ✔ No type needed // list.add(list.get(0)); // ✘ Not allowed }
List<?> vs List<T> — Side by Side
FeatureList<?>List<T>
Knows the type?No — unknown typeYes — exact type at call time
Read asObject onlyT directly
Add elementOnly nulllist.add(list.get(0)) ✔
Return typeCannot return TCan return T
Use caseRead-only, size/clearSwap, process, transform
// List<T> — type known, full operations allowed: public static <T> void swap(List<T> list, int i, int j) { T temp = list.get(i); list.set(i, list.get(j)); list.set(j, temp); // ✔ Works — T is known } // Cannot do this with List<?> — compiler doesn't know the type!
✦ T is decided at call time: fun(List<String>) → T = String  |  fun(List<Integer>) → T = Integer

☕ Type Erasure in Java
What is Type Erasure?
  • The process of removing generic type info at compile time.
  • JVM does not understand generics — compiler converts to non-generic code.
  • Ensures backward compatibility with pre-Java 5 code.
🔑 Key: No generic information exists at runtime — only at compile time.
How It Works — 3 Cases
// Case 1: Unbounded type class Box<T> { T value; } // After erasure: class Box { Object value; } // T → Object // Case 2: Bounded type class Box<T extends Number> { T value; } // After erasure: class Box { Number value; } // T → Number (upper bound) // Case 3: Generic list usage List<String> list = new ArrayList<>(); list.add("Hello"); // After erasure: List list = new ArrayList(); list.add("Hello");
COMPILE TIME | RUNTIME (JVM) ─────────────────────── | ────────────────────── Box<String> → type label | Box → no label (just Box) Box<Integer> → type label | Box → no label (just Box) | Compiler adds casts | Casts are already there: String s = list.get(0) | String s = (String)list.get(0)
Key Effects of Type Erasure

1. Method Overloading Restriction

void fun1(List<Integer> list) { } void fun1(List<String> list) { } // ✘ Compile Error! // After erasure: both become void fun1(List list) — same signature!

2. Compiler Inserts Casts Automatically

List<String> list = List.of("Java", "Python"); String s = list.get(0); // Internally becomes: String s = (String) list.get(0); // cast added by compiler
⚠ Generics = Compile-time safety ONLY.
At runtime, all type info is erased. JVM just sees raw types.
🎁 Gift Box Analogy:
At compile time → box has a label ("This is a String box")
At runtime → label is removed → just a box, contents are Object
