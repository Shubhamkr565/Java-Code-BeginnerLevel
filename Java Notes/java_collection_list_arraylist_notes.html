★ Autoboxing & Autounboxing java 1.5
Why needed? Collections store only Objects — not primitives. Before Java 1.5, developers had to manually convert. Java 1.5 introduced autoboxing (auto-wrapping) to do this silently.
❌ Before Java 1.5 (Manual)
int i = 10; // Boxing (manual) Integer i1 = Integer.valueOf(i); // Unboxing (manual) int x = i1.intValue();
✓ Java 1.5+ (Automatic)
int i = 10; // Autoboxing ↓ Integer i1 = i; // Autounboxing ↓ int x = i1;
8 Primitives → 8 Wrapper Classes:
int → Integer byte → Byte char → Character boolean → Boolean long → Long float → Float double → Double short → Short
★ Type Safety & Generics java 1.5
Problem with raw ArrayList: Any object can be added anywhere → must downcast every time → risk of ClassCastException at runtime!
❌ Type-Unsafe (No Generics)
ArrayList al = new ArrayList(); al.add(10); // int, ok al.add("hello"); // String, ok (no error!) // Must downcast every get: Integer x = (Integer) al.get(0); // ClassCastException if wrong!
✓ Type-Safe (With Generics)
ArrayList<String> al = new ArrayList<>(); al.add("delhi"); // ✓ OK al.add(10); // ✗ Compile error! // No downcast needed: String s = al.get(0); // Direct ✓
📌 Diamond Operator <> — skip repeating the type on the right side:
ArrayList<Student> al = new ArrayList<>();  ←  Java infers the type automatically

⚠️ ArrayList<Object> is equivalent to a type-unsafe collection — it defeats the purpose of generics!
How Generics Changed ArrayList Internals
Before Generics
public class ArrayList { public boolean add(Object obj){...} public Object get(int index){...} }
After Generics (Java 1.5+)
public class ArrayList<T> { public boolean add(T t){...} public T get(int index){...} }
