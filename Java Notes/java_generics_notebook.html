☕ Java Generics — Revisited
What are Generics?
  • Introduced in Java 5 — allows reusable, type-safe code.
  • Write one class / method that works with any type.
  • <T>, <E>, <K,V> are just placeholders for types.
  • No need to write separate classes for String, Integer, Employee, etc.
⚠ Important: Generics work only with Reference Types (String, Integer…). Primitives (int, double) are NOT allowed directly → use wrapper classes.
🔑 Key: Provides compile-time type safety — NOT runtime safety (due to type erasure).
Problem Without Generics
  • Had to use Object and do manual downcasting.
  • Caused: No type safety ClassCastException Messy code
Without Generics (error-prone): MyGen gen = new MyGen(); gen.add("Hello"); String s = (String) gen.get(); ← manual cast needed! gen.add(new Integer(5)); String bad = (String) gen.get(); ← ClassCastException at RUNTIME!
Solution — Generic Class
public class MyGen<T> { private T obj; void add(T obj) { this.obj = obj; } T get() { return obj; } }
// Usage — no casting needed! MyGen<String> g1 = new MyGen<>(); g1.add("Hello"); MyGen<Integer> g2 = new MyGen<>(); g2.add(20); String s = g1.get(); // ✔ exact type back, no cast

Bounded Types (Constraints)
  • Use extends keyword to limit which types are accepted.
UNBOUNDED BOUNDED (class) BOUNDED (interface) ───────────── ────────────────── ──────────────────────── class Test<T> class Test<T extends class Test<T extends Number> Runnable> ↓ accepts ↓ accepts ↓ accepts Any type Number or subtypes Runnable implementations (Integer, Float…) (classes that implement it)

✦ Example with class hierarchy:

Payment ← parent ↳ CashPayment ← child ↳ CardPayment ← child class Bill<T extends Payment> { } Bill<Payment> bill = new Bill<>(); // ✔ Bill<CashPayment> bill = new Bill<>(); // ✔ // Bill<Integer> bill = new Bill<>(); // ✘ Invalid
⚠ Rules for Bounded Types:
✘ implements keyword NOT allowed at class level — use extends for both class and interface.
✘ super keyword NOT allowed at class level for type parameters.
Multiple Bounds (using &)
  • Type must satisfy all conditions.
  • Syntax: <T extends ClassType & InterfaceType>
class Test<T extends Number & Serializable> { } // Integer → ✔ (subclass of Number AND implements Serializable) // String → ✘ (NOT a subclass of Number)
CombinationValid?Rule
<T extends Number & Thread>✘ InvalidTwo class types not allowed
<T extends Runnable & Serializable>✔ ValidTwo interface types — allowed
<T extends Number & Runnable>✔ ValidClass first, then interface ✔
<T extends Runnable & Number>✘ InvalidInterface before class — wrong order
✦ Order Rule: Always write Class before Interface when using &.

☕ Generic Methods
  • Can declare generic methods inside any class (generic or not).
  • Type parameter <E> must be declared before the return type.
Syntax: public static <E> void fun1(E e) { ... } ↑ ↑ type param return type (before return type!)

✦ Generic method — any type:

public static <E> void fun1(E e) { System.out.println(e); } // Works with any type: fun1(10); // Integer fun1("Hello"); // String fun1(new A()); // Custom class

✦ Generic method — key-value pair:

public static <K, V> void print(K key, V value) { System.out.println(key + "===" + value); } // print("Name", "Alice") or print(1, 99.5) — both work

✦ Generic printArray — one method for any array type:

public static <E> void printArray(E[] arr) { for(E e : arr) System.out.println(e); } Integer[] nums = {10, 20, 30}; String[] cities = {"Delhi", "Pune"}; printArray(nums); // ✔ printArray(cities); // ✔
⚠ Remember: Primitives not allowed in generic methods — use Integer[], not int[].
Hierarchy with Generic Methods: Employee[] → printArray(employees) ✔ Developer[] → printArray(developers) ✔ (Developer extends Employee) Manager[] → printArray(managers) ✔ (Manager extends Employee) int[] → printArray(ints) ✘ (primitives not allowed)
